# 电控组代码规范

## 零、注意事项

1. Venom战队电控组成员必须严格遵守该代码规范，若复查时发现不符合规范会被要求修改。
2. 目的：不是写封装复杂看起来很牛的代码，而是写逻辑清晰、层级分明的代码，让后续使用者能快速找到应该如何添加自己想要的功能。（湖南大学跃鹿的代码封装性太强，但是过于封装就导致当我不熟悉该代码框架而想使用很简单的功能时，或者想查找代码中是否使用了该功能时往往要找很久，反而是一种时间的浪费！！！）
3. 目的：该代码框架可以给所有陆地兵种使用（步兵，英雄，哨兵，工程）

## 一、命名规范

### 1. 文件命名

- 小写字母 + 下划线
- 不能与CUBEMX自动生成的文件名一致，若一致需要在文件名前加前缀以示区分
- 示例：`bsp_gpio.c`, `bsp_can.h`, `motor_control.c`, `system_monitor.h`

### 2. 文件夹命名

- 单词首字母大写，其余小写，不同单词之间使用下划线隔开
- 不能与CUBEMX自动生成的文件名一致，若一致需要在文件名前加前缀以示区分
- 示例：`Remote_Control`, `Gimbal_Task`, `Chassis_Task`

### 3. 变量命名

- 小写+下划线（snake_case）
- 变量名中不同类型的词语要用下划线隔开
- 示例：
  ```c
  uint32_t system_tick;      // 系统时钟
  static uint8_t error_count; // 错误计数器
  const float pid_kp = 1.5f; // PID比例常数
  ```

### 4. 函数命名

- 首字母大写，其余小写，单词间用下划线分隔
- 示例：
  ```c
  void Uart_init(UART_HandleTypeDef *huart);
  bool Motor_set_speed(uint8_t id, int16_t rpm);
  float Pid_update(Pid_controller *pid, float error);
  ```

### 5. 结构体/枚举命名

- 结构体加 `_t` 后缀，枚举类型加 `_e` 后缀
- 成员变量：snake_case
- 示例：
  ```c
  typedef struct {
      float kp;
      float ki;
      float kd;
      float integral;
  } Pid_controller_t;

  typedef enum {
      Motor_state_idle,
      Motor_state_running,
      Motor_state_error
  } Motor_state_e;
  ```

### 6. 宏定义

- 全大写 + 下划线
- 示例：
  ```c
  #define UART_TIMEOUT_MS  100
  #define MOTOR_MAX_RPM    3000
  ```

## 二、缩进/括号格式规范

### 7. 缩进与空格

- Tab 缩进
- 操作符两侧加空格
- 示例：
  ```c
  int32_t result = (a + b) * c;
  if (condition) {
      // ...
  }
  ```

### 8. 大括号风格

- K&R风格（左大括号不换行）
- 示例：
  ```c
  void Function_name(void) {
      // ...
  }

  if (condition) {
      // ...
  } else {
      // ...
  }
  ```

## 三、函数设计规范

### 9. 函数命名示例（模块+谓语动词+宾语）

- 首字母大写，其余小写
- 示例：
  ```c
  // 初始化函数
  void Uart_init(UART_HandleTypeDef *huart);
  void Gpio_config(GPIO_TypeDef *port, uint16_t pin);

  // 控制函数
  void Motor_set_target(uint8_t id, float target);
  void Pid_reset(Pid_controller_t *pid);

  // 状态获取
  bool System_is_ready(void);
  uint32_t Timer_get_value(void);
  ```

### 10. 参数设计

- 函数原型中不超过4个参数
- 复杂参数使用结构体代替
- 示例：
  ```c
  typedef struct {
      uint8_t id;
      int16_t target_rpm;
      uint32_t timeout_ms;
  } Motor_command_t;

  bool Motor_send_command(const Motor_command_t *cmd);
  ```

## 四、注释规范

### 1. 文件头注释（作者统一用队名VENOM）

```c
/**
 * @file    bsp_uart.c
 * @brief   UART硬件抽象层
 * @author  VENOM 
 * @date    2023-07-01
 * @version 1.0
 *
 * @note    支持DMA传输和中断接收
 *          波特率范围：9600-115200
 */
```

### 2. 函数注释（Doxygen风格）

```c
/**
 * @brief 初始化UART外设
 * @param huart UART句柄指针
 * @param baudrate 波特率 (9600, 115200等)
 * @return 初始化状态
 * @retval true  初始化成功
 * @retval false 初始化失败
 */
bool bsp_uart_init(UART_HandleTypeDef *huart, uint32_t baudrate);
```

### 3. 代码注释

- 解释该行代码的目的（Purpose），而不是做什么（What）
- 避免显而易见的注释
- 示例：
  ```c
  // 使用位带操作提高GPIO操作速度
  GPIO_BITBAND(GPIOA, 5) = 1;
  ```

## 五、数据类型规范

### 1. 使用标准类型

```c
#include <stdint.h>
#include <stdbool.h>
#include <stddef.h>

uint8_t  byte_data;
uint16_t sensor_value;
int32_t  position;
bool     is_ready;
```

### 2. 禁止原生类型

- ❌ 避免使用：int, long, short
- ✅ 使用：int32_t, uint16_t等

### 3. 结构体对齐

- 显式处理内存对齐
- 示例：
  ```c
  #pragma pack(push, 1)
  typedef struct {
      uint8_t header;
      uint16_t data;
      uint8_t checksum;
  } sensor_packet_t;
  #pragma pack(pop)
  ```

## 六、代码层级划分规范（初定）

### 1. 层级划分

- 所有兵种的工程代码必须采用分层级的框架设计，由低往高，高层调用底层接口，底层不允许触碰高层。
- 目的是为了工程逻辑清晰、可读性强，便于调试与后续开发新功能。
- 框架设计即将硬件驱动与功能实现、应用分开
- 示例：创建5个文件夹（初定，后续层级设计经讨论决定），分别命名为：

  ```
  0-Middleware_Layer
  1-Hardware_Layer
  2-Hardware_Driver_Layer
  3-Function_Module_Layer
  4-Application_Layer
  ```

- Middleware层（中间件层）：写算法的接口（各类滤波，控制算法等）、通信模型（Pub-Sub）等
- Hardware层（硬件抽象层）：对单片机内部设备（GPIO, TIM, UART, IIC, SPI, CAN等）的功能进行封装
- Hardware-Driver层（硬件驱动层）：对与单片机直接连接/被单片机控制的外部设备写驱动库，如陀螺仪BMI088、OLED、电机CAN驱动库、裁判系统通信驱动库等
- Function-Module-Layer层（方法模块层）：对各个模块的使用、功能进行编写，如底盘、云台的移动，发射机构的发射（单发、连发）、UI设计等。该层目的是让APP层只需调用该层几个接口函数即可实现功能，简化APP层代码量
- Application层（应用层）：机器人最终运动的执行层。若使用FreeRTOS，则分为多个任务task函数，在task函数里调用方法模块层接口实现功能；若使用裸机开发，则只需一个任务函数，里面调用方法模块层接口实现功能

> 其实APP层完全可以把Function层的任务也做了，但这样做的目的是让APP层任务函数只有几个功能函数的调用，看起来非常整洁易懂。如果APP层和FUN层合并，则会显得任务函数比较臃肿，可读性差了些。

## 七、全局变量定义规范

- 禁止使用外部全局变量，因为过多的外部全局变量会导致：
  1. 可读性差，合作者难以理解 
  2. 使用时需要记住该外部全局变量在哪里定义，且当文件多时容易出现重复命名，找bug非常麻烦
  3. 不专业，容易屎山

- 正确做法：
  1. 在本文件内实现的功能只使用静态全局变量，变量定义前加`static`
  2. 涉及多文件内变量传输时，尽量使用返回类型为该变量类型的函数，通过函数赋值变量的方式代替外部全局变量
  3. 使用Pub_Sub等通信模型，只需在其他文件订阅该话题即可实现通信
